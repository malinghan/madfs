# v4.0 — 元数据管理

---

## 1. 版本目标

在 v3.0 基础上，为每个上传的文件生成一个配套的 `.meta` 文件，记录原始文件名、大小、来源地址、MD5 等信息，并提供 `GET /meta?name=xxx` 接口查询。

**本版本解决的问题：**
- 上传后只有 UUID 文件名，原始文件名丢失了，无法还原
- 无法验证文件完整性（有没有损坏？）
- 多节点同步时，接收方需要知道文件的来源节点地址（v5.0 的前置条件）

**本版本不涉及：** 多节点同步，`.meta` 文件目前只在本节点使用。

---

## 2. 核心功能说明

### 元数据文件：数据文件的"影子文件"

> 类比：每个包裹（数据文件）旁边都贴着一张快递单（.meta 文件）。快递单记录了：原始包裹名称、重量、寄件地址、防伪验证码（MD5）。数据文件和元数据文件永远成对出现，存在同一个货架区（子目录）。

```
~/madfs/3a/
├── 3a9c1b2d-4e5f-....jpg        ← 数据文件（实际内容）
└── 3a9c1b2d-4e5f-....jpg.meta   ← 元数据文件（描述信息）
```

`.meta` 文件内容（JSON 格式）：
```json
{
  "name": "3a9c1b2d-4e5f-6789-abcd-ef0123456789.jpg",
  "originalFilename": "photo.jpg",
  "size": 102400,
  "downloadUrl": "http://localhost:8090/download",
  "tags": {
    "md5": "d41d8cd98f00b204e9800998ecf8427e"
  }
}
```

### FileMeta 模型字段说明

| 字段 | 类型 | 说明 | 用途 |
|------|------|------|------|
| `name` | String | UUID 文件名（含扩展名） | 定位文件 |
| `originalFilename` | String | 用户上传时的原始文件名 | 还原文件名 |
| `size` | long | 文件字节数 | 完整性校验、同步去重 |
| `downloadUrl` | String | 本节点的下载地址前缀 | 标识来源节点（v5.0 去重用） |
| `tags` | Map | 扩展标签，如 `md5` | 灵活扩展 |

### MD5：文件的"指纹"

> 类比：MD5 就像包裹的防伪验证码。寄件时计算一次，收件时再算一次，两个码一样说明包裹没有被篡改或损坏。

```
原始文件内容  ──MD5算法──►  d41d8cd98f00b204e9800998ecf8427e
                              （32位十六进制字符串）

特性：
- 相同内容 => 相同 MD5（确定性）
- 内容改变一个字节 => MD5 完全不同（敏感性）
- 无法从 MD5 反推原始内容（单向性）
```

### 上传流程（新增 meta 部分）

```
POST /upload
  │
  ├─ [v2.0] 生成 UUID 文件名
  ├─ [v2.0] 计算子目录
  ├─ [v2.0] file.transferTo(dest)    ← 写数据文件
  │
  ├─ [v4.0 新增] 构建 FileMeta 对象
  │      meta.name             = "3a9c....jpg"
  │      meta.originalFilename = "photo.jpg"
  │      meta.size             = file.getSize()
  │      meta.downloadUrl      = config.getDownloadUrl()
  │
  ├─ [v4.0 新增] autoMd5=true?
  │      ├─ Yes: md5 = DigestUtils.md5Hex(dest)
  │      │       meta.tags.put("md5", md5)
  │      └─ No:  跳过
  │
  └─ [v4.0 新增] FileUtils.writeMeta(dest, meta)
         └─ 写入 ~/madfs/3a/3a9c....jpg.meta

查询元数据:
GET /meta?name=3a9c....jpg
  │
  ├─ subdir = getSubdir("3a9c...") => "3a"
  ├─ metaFile = ~/madfs/3a/3a9c....jpg.meta
  └─ 读取并返回 JSON 内容
```

## 4. 代码实现说明

### 4.1 FileMeta — 元数据模型

```java
public class FileMeta {

    private String name;               // UUID 文件名: "3a9c....jpg"
    private String originalFilename;   // 原始文件名: "photo.jpg"
    private long size;                 // 文件大小（字节）
    private String downloadUrl;        // 本节点下载地址前缀: "http://localhost:8090/download"
    private Map<String, String> tags = new HashMap<>();  // 扩展标签，如 md5

    // 构造方法、getter/setter 省略
}
```

**关键点：** `downloadUrl` 存的是"下载地址前缀"而非完整 URL。
完整下载 URL = `downloadUrl + "?name=" + name`，
这样设计是为了让 v5.0 的 MQ 消费者能用 `downloadUrl` 识别消息来自哪个节点。

### 4.2 MadfsConfigProperties — 新增配置项

```java
@Component
@ConfigurationProperties(prefix = "madfs")
public class MadfsConfigProperties {

    private String path;
    private String downloadUrl;   // 本节点对外下载地址，如 "http://localhost:8090/download"
    private boolean autoMd5 = true;  // 是否自动计算并存储 MD5，默认开启

    // getter/setter 省略
}
```

### 4.3 FileUtils — 新增 writeMeta()

```java
// 需要引入 Jackson 依赖用于 JSON 序列化
// <dependency>
//   <groupId>com.fasterxml.jackson.core</groupId>
//   <artifactId>jackson-databind</artifactId>
// </dependency>
// （spring-boot-starter-web 已包含，无需单独引入）

private static final ObjectMapper MAPPER = new ObjectMapper();

/**
 * 将 FileMeta 序列化为 JSON，写入 {dataFile}.meta 文件
 *
 * 示例: dataFile = ~/madfs/3a/3a9c....jpg
 *       写入    = ~/madfs/3a/3a9c....jpg.meta
 */
public static void writeMeta(File dataFile, FileMeta meta) throws IOException {
    File metaFile = new File(dataFile.getAbsolutePath() + ".meta");
    String json = MAPPER.writeValueAsString(meta);
    Files.writeString(metaFile.toPath(), json);
    log.info("[FileUtils] 写入Meta文件: {}", metaFile.getAbsolutePath());
}

/**
 * 从 .meta 文件读取 FileMeta 对象
 */
public static FileMeta readMeta(File dataFile) throws IOException {
    File metaFile = new File(dataFile.getAbsolutePath() + ".meta");
    String json = Files.readString(metaFile.toPath());
    return MAPPER.readValue(json, FileMeta.class);
}
```

### 4.4 FileController — 修改 upload()，新增 meta()

```java
@RestController
@Slf4j
public class FileController {

    @Autowired
    private MadfsConfigProperties config;

    @PostMapping("/upload")
    public String upload(@RequestParam MultipartFile file) throws IOException {
        log.info("[UPLOAD] ===== 收到上传请求 =====");
        log.info("[UPLOAD] 原始文件名: {}, 大小: {} bytes",
                file.getOriginalFilename(), file.getSize());

        // [v2.0] 生成 UUID 文件名、计算子目录、写入文件
        String uuidName = FileUtils.getUUIDFile(file.getOriginalFilename());
        String subdir = FileUtils.getSubdir(uuidName);
        File dest = new File(config.getUploadPath() + "/" + subdir + "/" + uuidName);
        file.transferTo(dest);
        log.info("[UPLOAD] 文件写入完成: {}", dest.getAbsolutePath());

        // [v4.0 新增] 构建 FileMeta
        FileMeta meta = new FileMeta();
        meta.setName(uuidName);
        meta.setOriginalFilename(file.getOriginalFilename());
        meta.setSize(file.getSize());
        meta.setDownloadUrl(config.getDownloadUrl());

        // [v4.0 新增] 自动计算 MD5
        if (config.isAutoMd5()) {
            String md5 = DigestUtils.md5Hex(new FileInputStream(dest));
            meta.getTags().put("md5", md5);
            log.info("[UPLOAD] 计算MD5: {}", md5);
        }

        // [v4.0 新增] 写入 .meta 文件
        FileUtils.writeMeta(dest, meta);

        log.info("[UPLOAD] ===== 上传完成, 返回文件名: {} =====", uuidName);
        return uuidName;
    }

    // download() 方法保持不变...

    /**
     * 元数据查询接口
     * GET /meta?name=xxx
     * 返回对应的 .meta 文件内容（JSON 字符串）
     */
    @GetMapping("/meta")
    public String meta(@RequestParam String name) throws IOException {
        log.info("[META] 查询元数据, name={}", name);

        String subdir = FileUtils.getSubdir(name);
        File dataFile = new File(config.getUploadPath() + "/" + subdir + "/" + name);
        File metaFile = new File(dataFile.getAbsolutePath() + ".meta");

        log.info("[META] Meta文件路径: {}", metaFile.getAbsolutePath());

        if (!metaFile.exists()) {
            return "{}";  // 文件不存在返回空 JSON
        }

        String content = Files.readString(metaFile.toPath());
        log.info("[META] 返回内容: {}", content);
        return content;
    }
}
```

**关键点：** `DigestUtils.md5Hex()` 来自 `commons-codec`，
`spring-boot-starter-web` 间接依赖了它，通常无需单独引入。
注意要用 `new FileInputStream(dest)` 而非 `file.getInputStream()`，
因为 `transferTo()` 之后 `MultipartFile` 的临时文件可能已被清理。

## 5. 测试流程

### 5.1 基础元数据测试

```bash
# 准备测试文件
echo "Hello madfs meta" > /tmp/test-meta.txt

# 上传文件
FILENAME=$(curl -s -X POST http://localhost:8090/upload \
  -F "file=@/tmp/test-meta.txt")
echo "上传成功: $FILENAME"

# 查询元数据
curl -s "http://localhost:8090/meta?name=$FILENAME" | python3 -m json.tool
# 预期输出:
# {
#     "name": "3a9c1b2d-....txt",
#     "originalFilename": "test-meta.txt",
#     "size": 17,
#     "downloadUrl": "http://localhost:8090/download",
#     "tags": {
#         "md5": "xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx"
#     }
# }
```

### 5.2 验证 .meta 文件存在

```bash
SUBDIR="${FILENAME:0:2}"

# 查看子目录，应该有数据文件和 .meta 文件
ls -lh ~/madfs/$SUBDIR/
# 预期输出（两个文件成对出现）:
# -rw-r--r--  1 user  staff   17B  3a9c....txt
# -rw-r--r--  1 user  staff  180B  3a9c....txt.meta

# 直接查看 .meta 文件内容
cat ~/madfs/$SUBDIR/$FILENAME.meta
```

### 5.3 验证 MD5 正确性

```bash
# 上传文件并获取元数据中的 MD5
FILENAME=$(curl -s -X POST http://localhost:8090/upload \
  -F "file=@/tmp/test-meta.txt")

META_MD5=$(curl -s "http://localhost:8090/meta?name=$FILENAME" \
  | python3 -c "import sys,json; print(json.load(sys.stdin)['tags']['md5'])")
echo "元数据中的 MD5: $META_MD5"

# 本地计算 MD5
SUBDIR="${FILENAME:0:2}"
LOCAL_MD5=$(md5sum ~/madfs/$SUBDIR/$FILENAME | awk '{print $1}')
echo "本地计算的 MD5: $LOCAL_MD5"

# 比对
if [ "$META_MD5" = "$LOCAL_MD5" ]; then
  echo "MD5 一致 ✓"
else
  echo "MD5 不一致 ✗"
fi
```

### 5.4 验证原始文件名保留

```bash
# 上传一个有特殊名字的文件
echo "content" > /tmp/我的照片-2024.txt
FILENAME=$(curl -s -X POST http://localhost:8090/upload \
  -F "file=@/tmp/我的照片-2024.txt")

# 查询元数据，验证原始文件名
curl -s "http://localhost:8090/meta?name=$FILENAME" \
  | python3 -c "import sys,json; d=json.load(sys.stdin); print('原始文件名:', d['originalFilename'])"
# 预期: 原始文件名: 我的照片-2024.txt
```

### 5.5 完整验证脚本

```bash
#!/bin/bash
set -e

BASE_URL="http://localhost:8090"

echo "=== v4.0 元数据功能验证 ==="

# 准备测试文件
echo "madfs meta test" > /tmp/v4test.txt
ORIGINAL_SIZE=$(wc -c < /tmp/v4test.txt)

# 上传
echo "1. 上传文件..."
FILENAME=$(curl -s -X POST $BASE_URL/upload -F "file=@/tmp/v4test.txt")
echo "   文件名: $FILENAME"

# 查询元数据
echo "2. 查询元数据..."
META=$(curl -s "$BASE_URL/meta?name=$FILENAME")
echo "   元数据: $META"

# 验证各字段
echo "3. 验证元数据字段..."

ORIG_NAME=$(echo $META | python3 -c "import sys,json; print(json.load(sys.stdin)['originalFilename'])")
META_SIZE=$(echo $META | python3 -c "import sys,json; print(json.load(sys.stdin)['size'])")
META_MD5=$(echo $META | python3 -c "import sys,json; print(json.load(sys.stdin)['tags']['md5'])")

[ "$ORIG_NAME" = "v4test.txt" ] && echo "   originalFilename 正确 ✓" || echo "   originalFilename 错误 ✗: $ORIG_NAME"
[ "$META_SIZE" = "$ORIGINAL_SIZE" ] && echo "   size 正确 ✓" || echo "   size 错误 ✗: $META_SIZE vs $ORIGINAL_SIZE"
[ -n "$META_MD5" ] && echo "   md5 存在 ✓: $META_MD5" || echo "   md5 缺失 ✗"

# 验证 .meta 文件存在
SUBDIR="${FILENAME:0:2}"
[ -f "$HOME/madfs/$SUBDIR/$FILENAME.meta" ] \
  && echo "   .meta 文件存在 ✓" \
  || echo "   .meta 文件不存在 ✗"

echo ""
echo "=== v4.0 验证通过 ✓ ==="
```

## 6. 版本总结

### 学到了什么

| 知识点 | 说明 |
|--------|------|
| 伴随文件模式 | 数据文件 + 元数据文件成对存储，是文件系统中常见的设计模式 |
| Jackson ObjectMapper | Spring Boot 内置的 JSON 序列化/反序列化工具 |
| MD5 用途 | 文件完整性校验，不是加密，不能防篡改，只能检测意外损坏 |
| `downloadUrl` 的前瞻设计 | 当前版本只是记录，v5.0 会用它做节点去重 |

### 与 v3.0 对比

| 对比项 | v3.0 | v4.0 |
|--------|------|------|
| 原始文件名 | 丢失 | 保存在 .meta 中 |
| 文件大小 | 只能从文件系统查 | 记录在 .meta 中 |
| 完整性校验 | 无 | MD5 指纹 |
| 元数据查询接口 | 无 | GET /meta |
| 存储文件数量 | 每个文件 1 个 | 每个文件 2 个（数据 + meta） |

### 局限性

- `.meta` 文件和数据文件没有原子性保证：如果写完数据文件后进程崩溃，`.meta` 文件不存在，会出现"孤儿文件"
- MD5 计算需要再次读取整个文件，对大文件有额外 IO 开销（可优化为上传时流式计算）
- 元数据只存在本地，多节点场景下各节点的 `.meta` 文件需要单独同步（v5.0 处理）

### 设计要点

**为什么用 `.meta` 文件而不是数据库？**

- 无需额外依赖（数据库服务），部署更简单
- 元数据和数据文件在同一目录，迁移/备份时不会分离
- 文件系统本身就是一种 KV 存储，`filename.meta` 就是 key，JSON 内容就是 value
- 代价是：无法做复杂查询（如"查找所有大于 1MB 的文件"），v7.0 的 `/list` 接口需要遍历目录

**`tags: Map<String, String>` 的扩展性：**

用 Map 而非固定字段存储扩展属性，未来可以不改模型结构就加入新属性：
```json
{
  "tags": {
    "md5": "d41d...",
    "sha256": "e3b0...",    // 未来可加
    "width": "1920",        // 图片宽度
    "height": "1080"        // 图片高度
  }
}
```
