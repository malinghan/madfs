# v5.0 — MQ 异步同步（多节点广播）

---

## 1. 版本目标

在 v4.0 基础上，引入 RocketMQ，实现多节点间的文件异步同步。节点 A 上传文件后，通过 MQ 广播文件元数据，其他节点收到消息后自动从源节点拉取文件，最终所有节点都拥有相同的文件副本。

**本版本解决的问题：**
- 单节点存储有单点故障风险，需要多节点冗余
- 节点间如何感知"有新文件需要同步"？（MQ 广播）
- 如何避免节点同步自己发出的消息？（downloadUrl 去重）
- 如何保证重复消费不会重复下载？（幂等性）

**本版本不涉及：** 同步 HTTP 推送（v6.0），节点自动发现（v10.0）。

---

## 2. 核心功能说明

### 整体类比：广播电台调货

> 类比：A 仓库入库新包裹后，通过广播电台（RocketMQ）播报："我这里有个新包裹，单号 3a9c，需要的仓库来取"。
> - 所有仓库（包括 A 自己）都收到广播
> - A 仓库听到后发现是自己播的，直接忽略
> - B、C 仓库听到后，各自去 A 仓库取货（HTTP 下载）
> - 如果 B 仓库已经有这个包裹了，就不用再取（幂等）

### MQ 在这里扮演什么角色？

```
没有 MQ 的世界:
  节点A 上传完成
    │
    └─ 直接 HTTP 推送给节点B、C、D...
         问题: 节点A 需要知道所有其他节点的地址
               某个节点宕机，推送失败怎么办？

有 MQ 的世界:
  节点A 上传完成
    │
    └─ 发消息到 MQ（只需知道 MQ 地址）
         │
         MQ 负责分发给所有订阅者
         │
         ├─ 节点B 自己来消费（拉模式）
         ├─ 节点C 自己来消费
         └─ 节点D 宕机了？消息在 MQ 里等着，恢复后再消费
```

MQ 解耦了生产者（上传节点）和消费者（同步节点），生产者不需要知道有多少消费者。

### 消息内容：发什么？

发送的不是文件本身，而是文件的"元数据"（FileMeta JSON）：

```json
{
  "name": "3a9c1b2d-4e5f-....jpg",
  "originalFilename": "photo.jpg",
  "size": 102400,
  "downloadUrl": "http://localhost:8090/download",
  "tags": { "md5": "d41d..." }
}
```

接收方拿到元数据后，再主动去 `downloadUrl?name=xxx` 下载文件。
这是"推元数据 + 拉文件"的模式，比直接在 MQ 里传文件二进制更合理（MQ 不适合传大消息）。

### 去重机制：防止自同步

```
每个节点有自己的 downloadUrl（身份标识）:
  节点A: downloadUrl = "http://localhost:8090/download"
  节点B: downloadUrl = "http://localhost:8091/download"

消息中的 downloadUrl 就是发送方的身份:
  节点A 发的消息: { "downloadUrl": "http://localhost:8090/download", ... }

消费时的判断:
  if (localUrl.equals(message.downloadUrl)) {
      // 消息是我自己发的，跳过
  }
```

### 幂等机制：防止重复下载

```
消费者收到消息后，下载前先检查:

检查1: .meta 文件是否已存在？
  if (metaFile.exists()) {
      // 已有元数据，跳过写入
  }

检查2: 数据文件是否已存在且大小一致？
  if (file.exists() && file.length() == meta.getSize()) {
      // 文件已完整，跳过下载
  }

这样即使消息被重复投递，也不会重复下载文件。
```

### 完整同步流程

```
节点A: 上传完成
  │
  ▼
MQSyncer.sync(meta)
  │
  └─ rocketMQTemplate.send("madfs", JSON(meta))
          │
          ▼
    RocketMQ Broker
    Topic: madfs
          │
    ┌─────┴──────┐
    ▼            ▼
节点A 消费者   节点B 消费者
    │            │
    ▼            ▼
解析 FileMeta  解析 FileMeta
    │            │
localUrl        localUrl
== downloadUrl  != downloadUrl
    │            │
    ▼            ▼
  忽略         处理同步
               │
               ├─ 写 .meta 文件（幂等）
               │
               ├─ 检查本地文件（幂等）
               │
               └─ GET http://节点A:8090/download?name=3a9c....jpg
                      │
                      ▼
                  写入 /tmp/madfs-b/3a/3a9c....jpg
```

```java
    private String downloadUrl;
    private boolean autoMd5 = true;
    private String group;        // RocketMQ 消费者组，每个节点必须不同，如 "C8090"、"C8091"

    // getter/setter 省略
}
```

**关键点：** 每个节点的 `group` 必须不同。RocketMQ 的广播模式下，
同一个 group 内的消费者共享消息（只有一个消费者收到），
不同 group 各自独立消费（每个 group 都收到一份）。
我们需要每个节点都收到消息，所以每个节点用不同的 group。

### 4.3 FileUtils — 新增 download()

```java
/**
 * 从远端 URL 下载文件到本地
 * 使用 16KB 缓冲区流式写入，避免大文件占用大量内存
 *
 * @param url   远端下载地址，如 "http://localhost:8090/download?name=3a9c....jpg"
 * @param dest  本地目标文件
 */
public static void download(String url, File dest) throws IOException {
    log.info("[FileUtils] 从远端下载文件: {} => {}", url, dest.getAbsolutePath());

    // 确保父目录存在
    dest.getParentFile().mkdirs();

    URL remoteUrl = new URL(url);
    byte[] buffer = new byte[16 * 1024];
    int bytesRead;

    try (InputStream in = remoteUrl.openStream();
         FileOutputStream out = new FileOutputStream(dest)) {
        while ((bytesRead = in.read(buffer)) != -1) {
            out.write(buffer, 0, bytesRead);
        }
    }

    log.info("[FileUtils] 下载完成: {}, 大小: {} bytes",
            dest.getAbsolutePath(), dest.length());
}
```

### 4.4 MQSyncer — 生产者 + 消费者

```java
@Component
@Slf4j
public class MQSyncer {

    private static final String TOPIC = "madfs";

    @Autowired
    private RocketMQTemplate rocketMQTemplate;

    @Autowired
    private MadfsConfigProperties config;

    private static final ObjectMapper MAPPER = new ObjectMapper();

    /**
     * 发送文件元数据到 MQ，通知其他节点同步
     */
    public void sync(FileMeta meta) {
        try {
            String json = MAPPER.writeValueAsString(meta);
            log.info("[MQ-SYNC] ===== 发送MQ消息 =====");
            log.info("[MQ-SYNC] Topic: {}, Payload: {}", TOPIC, json);

            rocketMQTemplate.send(TOPIC, MessageBuilder.withPayload(json).build());

            log.info("[MQ-SYNC] 消息发送完成");
        } catch (Exception e) {
            log.error("[MQ-SYNC] 消息发送失败", e);
        }
    }

    /**
     * MQ 消费者：接收其他节点发来的文件元数据，下载文件到本地
     *
     * 注意: @RocketMQMessageListener 的 consumerGroup 需要动态配置
     * 这里用 SpEL 表达式从配置文件读取
     */
    @Component
    @RocketMQMessageListener(
        topic = TOPIC,
        consumerGroup = "${madfs.group}"   // 每个节点配置不同的 group
    )
    public class FileMQSyncer implements RocketMQListener<String> {

        @Override
        public void onMessage(String message) {
            log.info("[MQ-CONSUMER] ===== 收到MQ消息 =====");
            log.info("[MQ-CONSUMER] 消息体: {}", message);

            try {
                // 1. 反序列化元数据
                FileMeta meta = MAPPER.readValue(message, FileMeta.class);
                log.info("[MQ-CONSUMER] 解析FileMeta: name={}, size={}, downloadUrl={}",
                        meta.getName(), meta.getSize(), meta.getDownloadUrl());

                // 2. 去重：跳过本机发出的消息
                String localUrl = config.getDownloadUrl();
                if (localUrl.equals(meta.getDownloadUrl())) {
                    log.info("[MQ-CONSUMER] 消息来自本机, 忽略 (防止自同步)");
                    return;
                }
                log.info("[MQ-CONSUMER] 消息来自其他节点, 开始处理同步");

                // 3. 计算本地存储路径
                String subdir = FileUtils.getSubdir(meta.getName());
                File dataFile = new File(config.getUploadPath() + "/" + subdir + "/" + meta.getName());
                File metaFile = new File(dataFile.getAbsolutePath() + ".meta");

                // 4. 幂等写入 .meta 文件
                if (!metaFile.exists()) {
                    String metaJson = MAPPER.writeValueAsString(meta);
                    Files.writeString(metaFile.toPath(), metaJson);
                    log.info("[MQ-CONSUMER] Meta文件写入完成: {}", metaFile.getAbsolutePath());
                } else {
                    log.info("[MQ-CONSUMER] Meta文件已存在, 跳过");
                }

                // 5. 幂等下载文件
                if (dataFile.exists() && dataFile.length() == meta.getSize()) {
                    log.info("[MQ-CONSUMER] 文件已存在且大小一致, 跳过下载");
                    return;
                }

                // 6. 从源节点下载文件
                String downloadUrl = meta.getDownloadUrl() + "?name=" + meta.getName();
                log.info("[MQ-CONSUMER] 开始从源节点下载: {}", downloadUrl);
                FileUtils.download(downloadUrl, dataFile);
                log.info("[MQ-CONSUMER] ===== 文件同步完成: {} =====", dataFile.getAbsolutePath());

            } catch (Exception e) {
                log.error("[MQ-CONSUMER] 消息处理失败", e);
            }
        }
    }
}
```

## 6. 版本总结

### 学到了什么

| 知识点 | 说明 |
|--------|------|
| 消息队列解耦 | 生产者只需知道 MQ 地址，不需要知道有多少消费者 |
| 广播 vs 集群消费 | 不同 group 各自独立消费 = 广播；同一 group 共享消费 = 负载均衡 |
| 推元数据 + 拉文件 | MQ 传轻量元数据，文件通过 HTTP 拉取，避免 MQ 传大消息 |
| 幂等设计 | 消费者必须设计为幂等，因为 MQ 可能重复投递消息 |
| 最终一致性 | 上传成功时其他节点还没有文件，但最终会同步到，这是分布式系统的常见权衡 |

### 与 v4.0 对比

| 对比项 | v4.0 | v5.0 |
|--------|------|------|
| 节点数量 | 单节点 | 多节点 |
| 文件冗余 | 无 | 所有节点各有一份副本 |
| 同步方式 | 无 | MQ 异步广播 |
| 一致性 | 强一致（单节点） | 最终一致（异步同步） |
| 外部依赖 | 无 | RocketMQ |

### 局限性

- 异步同步有延迟（通常 1~3 秒），上传后立即从其他节点下载可能还没同步完
- 节点 B 宕机期间的消息，恢复后能否消费取决于 RocketMQ 的消息保留策略
- 每个节点都存一份完整副本，存储空间利用率低（N 个节点存 N 份）
- 没有同步状态监控，不知道某个节点是否同步滞后

### 设计要点

**为什么用"推元数据 + 拉文件"而不是直接在 MQ 里传文件？**

RocketMQ 默认单条消息大小限制为 4MB，不适合传大文件。
而且 MQ 的职责是"通知"，不是"传输"。
让接收方主动拉取文件，还有一个好处：接收方可以控制拉取速度，不会被大量消息压垮。

**`consumerGroup` 为什么每个节点要不同？**

RocketMQ 的消费模型：
- 同一个 group 内的消费者：消息被其中一个消费（负载均衡，适合任务队列）
- 不同 group 的消费者：每个 group 都收到完整的消息（广播，适合数据同步）

我们需要每个节点都同步文件，所以每个节点用不同的 group，
这样每个节点都能独立消费到所有消息。
