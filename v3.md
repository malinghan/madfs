# v3.0 — 文件下载接口

---

## 1. 版本目标

在 v2.0 基础上，实现 `GET /download?name=xxx` 接口，根据文件名从本地存储找到文件，推断 MIME 类型，以流式方式输出给客户端。

**本版本解决的问题：**
- 客户端如何通过 HTTP 取回已上传的文件？
- 如何告诉浏览器文件是什么类型（图片/文本/视频）？
- 大文件如何避免一次性加载到内存？（流式输出）

**本版本不涉及：** 元数据查询、多节点同步。

---

## 2. 核心功能说明

### 流式输出：为什么不直接读到内存再返回？

> 类比：快递员取件时，不会把整个仓库的货物搬到办公室再交给客户，而是直接从货架取出包裹递给客户。文件下载也一样——用一个 16KB 的"小推车"（缓冲区），每次从磁盘搬一点，直接推给客户端，不在内存里堆积。

```
磁盘文件 (100MB)
    │
    │  每次读 16KB
    ▼
[缓冲区 16KB]  ──────────────────►  客户端
    │
    │  读完再读下一块
    ▼
[缓冲区 16KB]  ──────────────────►  客户端
    │
    ...（重复直到文件读完）
```

如果一次性读入内存：
- 100MB 文件 → 占用 100MB JVM 堆内存
- 100 个并发下载 → 占用 10GB 内存，直接 OOM

流式输出：
- 无论文件多大，内存占用始终只有 16KB（缓冲区大小）

### MIME 类型：告诉浏览器如何处理文件

> 类比：快递包裹上贴的"易碎品"、"食品"标签，告诉收件人如何处理这个包裹。HTTP 响应头的 `Content-Type` 就是这个标签，告诉浏览器收到的是图片、文本还是视频。

```
文件扩展名  =>  Content-Type
.jpg        =>  image/jpeg
.png        =>  image/png
.gif        =>  image/gif
.txt        =>  text/plain
.html       =>  text/html
.pdf        =>  application/pdf
.mp4        =>  video/mp4
.zip        =>  application/zip
其他        =>  application/octet-stream  (通用二进制，浏览器会触发下载)
```

### 下载完整流程

```
客户端
  │
  │  GET /download?name=3a9c1b2d-....jpg
  │
  ▼
FileController.download(name, response)
  │
  ├─ 1. 计算子目录
  │      subdir = getSubdir("3a9c...") => "3a"
  │
  ├─ 2. 构建文件路径
  │      file = ~/madfs/3a/3a9c1b2d-....jpg
  │
  ├─ 3. 推断 MIME 类型
  │      mimeType = getMimeType("3a9c....jpg") => "image/jpeg"
  │
  ├─ 4. 设置响应头
  │      Content-Type: image/jpeg
  │      Content-Length: 102400
  │
  └─ 5. 流式输出
         FileUtils.output(file, response.getOutputStream())
           │
           └─ while (read 16KB from file) {
                  write 16KB to outputStream
              }
  │
  ▼
客户端收到完整文件内容
```

## 4. 代码实现说明

### 4.1 FileUtils — 新增 getMimeType() 和 output()

```java
public class FileUtils {

    // v1.0 init(), v2.0 getUUIDFile(), getSubdir() 保持不变...

    /**
     * 根据文件名推断 MIME 类型
     * 取文件扩展名，映射到对应的 Content-Type
     */
    public static String getMimeType(String filename) {
        if (filename == null) return "application/octet-stream";

        String lower = filename.toLowerCase();

        if (lower.endsWith(".jpg") || lower.endsWith(".jpeg")) return "image/jpeg";
        if (lower.endsWith(".png"))  return "image/png";
        if (lower.endsWith(".gif"))  return "image/gif";
        if (lower.endsWith(".webp")) return "image/webp";
        if (lower.endsWith(".txt"))  return "text/plain";
        if (lower.endsWith(".html") || lower.endsWith(".htm")) return "text/html";
        if (lower.endsWith(".css"))  return "text/css";
        if (lower.endsWith(".js"))   return "application/javascript";
        if (lower.endsWith(".json")) return "application/json";
        if (lower.endsWith(".xml"))  return "application/xml";
        if (lower.endsWith(".pdf"))  return "application/pdf";
        if (lower.endsWith(".zip"))  return "application/zip";
        if (lower.endsWith(".mp4"))  return "video/mp4";
        if (lower.endsWith(".mp3"))  return "audio/mpeg";

        // 未知类型：通用二进制流，浏览器会触发下载
        return "application/octet-stream";
    }

    /**
     * 流式输出文件内容到 OutputStream
     * 使用 16KB 缓冲区，避免大文件占用大量内存
     *
     * @param file  要输出的文件
     * @param out   目标输出流（通常是 HttpServletResponse.getOutputStream()）
     */
    public static void output(File file, OutputStream out) throws IOException {
        byte[] buffer = new byte[16 * 1024];  // 16KB 缓冲区
        int bytesRead;

        try (FileInputStream fis = new FileInputStream(file)) {
            while ((bytesRead = fis.read(buffer)) != -1) {
                out.write(buffer, 0, bytesRead);
            }
        }
        // out 由调用方（Spring）负责关闭，这里不关闭
    }
}
```

**关键点：** `out.write(buffer, 0, bytesRead)` 中第三个参数 `bytesRead` 很重要——
文件最后一块可能不足 16KB，必须只写实际读到的字节数，否则会在文件末尾写入垃圾数据。

### 4.2 FileController — 新增 download()

```java
@RestController
@Slf4j
public class FileController {

    // upload() 方法保持不变...

    /**
     * 文件下载接口
     * GET /download?name=xxx
     * 直接向 HttpServletResponse 写入文件内容
     */
    @GetMapping("/download")
    public void download(@RequestParam String name,
                         HttpServletResponse response) throws IOException {
        log.info("[DOWNLOAD] ===== 收到下载请求, name={} =====", name);

        // 1. 定位文件
        String subdir = FileUtils.getSubdir(name);
        File file = new File(config.getUploadPath() + "/" + subdir + "/" + name);
        log.info("[DOWNLOAD] 文件路径: {}, 存在: {}, 大小: {} bytes",
                file.getAbsolutePath(), file.exists(), file.length());

        // 2. 文件不存在时返回 404
        if (!file.exists()) {
            response.setStatus(HttpServletResponse.SC_NOT_FOUND);
            return;
        }

        // 3. 设置响应头
        String mimeType = FileUtils.getMimeType(name);
        response.setContentType(mimeType);
        response.setContentLengthLong(file.length());
        log.info("[DOWNLOAD] MIME类型: {}", mimeType);

        // 4. 流式输出文件内容
        FileUtils.output(file, response.getOutputStream());
        log.info("[DOWNLOAD] ===== 文件输出完成 =====");
    }
}
```

**关键点：** 下载接口返回类型是 `void`，直接操作 `HttpServletResponse`，
而不是返回 `ResponseEntity<byte[]>`（后者会把整个文件加载到内存）。

## 5. 测试流程

### 5.1 基础下载测试

```bash
# 先上传一个文件
echo "Hello madfs download" > /tmp/test-download.txt
FILENAME=$(curl -s -X POST http://localhost:8090/upload \
  -F "file=@/tmp/test-download.txt")
echo "上传成功: $FILENAME"

# 下载文件
curl -o /tmp/downloaded.txt \
  "http://localhost:8090/download?name=$FILENAME"

# 验证内容一致
diff /tmp/test-download.txt /tmp/downloaded.txt \
  && echo "文件内容一致 ✓" \
  || echo "文件内容不一致 ✗"
```

### 5.2 验证响应头

```bash
# 上传文本文件，验证 Content-Type
FILENAME=$(curl -s -X POST http://localhost:8090/upload \
  -F "file=@/tmp/test-download.txt")

curl -I "http://localhost:8090/download?name=$FILENAME"
# 预期响应头:
# HTTP/1.1 200
# Content-Type: text/plain
# Content-Length: 21

# 上传图片，验证 MIME 类型（如果有图片的话）
# FILENAME=$(curl -s -X POST http://localhost:8090/upload -F "file=@/tmp/photo.jpg")
# curl -I "http://localhost:8090/download?name=$FILENAME"
# 预期: Content-Type: image/jpeg
```

### 5.3 验证 404 处理

```bash
# 下载不存在的文件
curl -v "http://localhost:8090/download?name=nonexistent.txt" 2>&1 | grep "< HTTP"
# 预期: < HTTP/1.1 404
```

### 5.4 验证大文件下载

```bash
# 生成 10MB 测试文件
dd if=/dev/urandom of=/tmp/large-file.bin bs=1024 count=10240

# 上传
FILENAME=$(curl -s -X POST http://localhost:8090/upload \
  -F "file=@/tmp/large-file.bin")
echo "上传大文件: $FILENAME"

# 下载并计时
time curl -o /tmp/large-downloaded.bin \
  "http://localhost:8090/download?name=$FILENAME"

# 验证文件大小一致
ls -lh /tmp/large-file.bin /tmp/large-downloaded.bin
# 预期: 两个文件大小相同（10MB）

# 验证内容一致（MD5 比对）
md5sum /tmp/large-file.bin /tmp/large-downloaded.bin
# 预期: 两行 MD5 值相同
```

### 5.5 完整上传-下载往返测试

```bash
#!/bin/bash
set -e

BASE_URL="http://localhost:8090"

echo "=== v3.0 上传-下载往返验证 ==="

# 准备不同类型的测试文件
echo "text content" > /tmp/v3-test.txt

# 上传
echo "1. 上传文件..."
FILENAME=$(curl -s -X POST $BASE_URL/upload -F "file=@/tmp/v3-test.txt")
echo "   文件名: $FILENAME"

# 验证响应头
echo "2. 验证响应头..."
CONTENT_TYPE=$(curl -sI "$BASE_URL/download?name=$FILENAME" \
  | grep -i "content-type" | awk '{print $2}' | tr -d '\r')
echo "   Content-Type: $CONTENT_TYPE"
if [[ "$CONTENT_TYPE" == *"text/plain"* ]]; then
  echo "   MIME 类型正确 ✓"
else
  echo "   MIME 类型错误 ✗"
fi

# 下载并验证内容
echo "3. 下载并验证内容..."
curl -s -o /tmp/v3-downloaded.txt "$BASE_URL/download?name=$FILENAME"
diff /tmp/v3-test.txt /tmp/v3-downloaded.txt \
  && echo "   内容一致 ✓" \
  || echo "   内容不一致 ✗"

echo ""
echo "=== v3.0 验证通过 ✓ ==="
```

## 6. 版本总结

### 学到了什么

| 知识点 | 说明 |
|--------|------|
| 流式 IO | 用固定大小缓冲区读写，内存占用与文件大小无关 |
| `HttpServletResponse` | 直接操作响应对象，比返回 `ResponseEntity<byte[]>` 更节省内存 |
| `Content-Type` | 告诉浏览器如何处理响应体，影响是否内联显示还是触发下载 |
| `Content-Length` | 让客户端知道文件总大小，可以显示下载进度 |

### 与 v2.0 对比

| 对比项 | v2.0 | v3.0 |
|--------|------|------|
| 上传 | ✓ | ✓ |
| 下载 | ✗（只能直接访问文件系统） | ✓ HTTP 下载 |
| MIME 类型 | 无 | 根据扩展名自动推断 |
| 大文件支持 | 上传有限制 | 下载流式，无内存压力 |
| 完整性验证 | 无 | 可通过 MD5 比对验证 |

### 局限性

- 不支持 HTTP Range 请求（断点续传），大文件下载中断后需重新下载（v8.0 解决）
- MIME 类型靠扩展名推断，不够准确（如文件内容是图片但扩展名是 `.txt`）
- 没有访问权限控制，任何人知道文件名就能下载
- 没有记录原始文件名，下载时文件名是 UUID 格式，不友好

### 设计要点

**为什么缓冲区选 16KB？**

16KB 是一个经验值，平衡了以下因素：
- 太小（如 1KB）：系统调用次数多，CPU 开销大
- 太大（如 1MB）：每个下载连接占用 1MB 内存，100 并发就是 100MB
- 16KB 接近操作系统的页大小（4KB）的整数倍，IO 效率较好

**`Content-Length` 的作用：**

设置 `Content-Length` 后，客户端（如 curl、浏览器）可以：
1. 显示下载进度百分比
2. 预分配接收缓冲区
3. 检测传输是否完整（收到的字节数 == Content-Length）

如果不设置，客户端只能等连接关闭才知道文件传完了。
