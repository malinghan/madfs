# v6.0 — HTTP 同步备份（主从直连）

---

## 1. 版本目标

在 v5.0 基础上，新增 HTTP 同步备份模式。当 `syncBackup=true` 时，上传完成后立即通过 HTTP 将文件推送到备份节点，实现同步复制；如果推送失败，自动降级为 MQ 异步补偿。

**本版本解决的问题：**
- v5.0 的 MQ 异步同步有延迟（1~3 秒），某些场景需要实时同步
- 主从架构下，希望上传成功时备份节点已经有文件了（强一致性）
- 备份节点宕机时，不能让上传失败，需要降级方案

**本版本不涉及：** 多备份节点（只支持一个 backupUrl），自动故障切换。

---

## 2. 核心功能说明

### 同步 vs 异步：两种同步模式对比

> 类比：
> - **MQ 异步同步**（v5.0）：快递员把包裹放在 A 仓库后，通过广播电台通知其他仓库"来取货"，然后就走了。其他仓库什么时候取到货，快递员不管。
> - **HTTP 同步备份**（v6.0）：快递员把包裹放在 A 仓库后，亲自开车送一份到 B 仓库，等 B 仓库签收了才离开。如果 B 仓库关门了，快递员改用广播电台通知（降级）。

```
┌─────────────────────────────────────────────────────────────┐
│                    上传完成后的同步策略                      │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  syncBackup=false (v5.0 默认)                                │
│    │                                                         │
│    └─► MQSyncer.sync(meta)                                  │
│           └─► 发送 MQ 消息，异步广播                         │
│                  └─► 其他节点自行消费下载                    │
│                                                              │
│  syncBackup=true (v6.0 新增)                                 │
│    │                                                         │
│    └─► HttpSyncer.sync(file, backupUrl)                     │
│           │                                                  │
│           ├─► 成功 ──► 返回（备份节点已有文件）              │
│           │                                                  │
│           └─► 失败 ──► MQSyncer.sync(meta)  (降级补偿)       │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 防止同步循环：X-Filename 头的作用

```
问题: 如果不加标识，会发生什么？

节点A 上传文件
  │
  └─► HttpSyncer 推送到节点B
         │
         └─► 节点B 收到 POST /upload
                │
                └─► 节点B 也触发 HttpSyncer 推送到节点A？
                       │
                       └─► 节点A 又推送到节点B？
                              │
                              └─► 无限循环！

解决方案: 用请求头标识"这是同步请求"

节点A 推送时设置:
  X-Filename: 3a9c1b2d-....jpg        (UUID 文件名)
  X-Orig-Filename: photo.jpg          (原始文件名)

节点B 检测到 X-Filename 存在:
  => 这是同步请求，不是用户上传
  => needSync = false（不再触发二次同步）
  => 直接存储文件，返回
```

### HTTP 同步完整流程

```
节点A: 用户上传完成
  │
  ├─ [v2.0~v4.0] 写数据文件、写 .meta 文件
  │
  ├─ [v6.0] syncBackup=true?
  │      │
  │      ├─ Yes ──► HttpSyncer.sync(file, backupUrl, originalFilename)
  │      │            │
  │      │            ├─ 构建 MultipartFile
  │      │            │   Header: X-Filename = "3a9c....jpg"
  │      │            │   Header: X-Orig-Filename = "photo.jpg"
  │      │            │   Body: file binary
  │      │            │
  │      │            ├─ POST http://localhost:8091/upload
  │      │            │       │
  │      │            │       ▼
  │      │            │  节点B: FileController.upload()
  │      │            │       │
  │      │            │       ├─ 检测到 X-Filename 不为空
  │      │            │       ├─ filename = X-Filename
  │      │            │       ├─ needSync = false
  │      │            │       └─ 直接存储，不触发同步
  │      │            │
  │      │            ├─ 成功 ──► log "HTTP同步成功"
  │      │            │
  │      │            └─ 失败 ──► MQSyncer.sync(meta)  (降级)
  │      │
  │      └─ No ──► MQSyncer.sync(meta)  (v5.0 逻辑)
  │
  └─ 返回文件名
```

### 4.2 HttpSyncer — HTTP 同步推送

```java
@Component
@Slf4j
public class HttpSyncer {

    @Autowired
    private RestTemplate restTemplate;

    /**
     * 通过 HTTP 将文件同步推送到备份节点
     *
     * @param file              本地文件
     * @param backupUrl         备份节点上传地址
     * @param originalFilename  原始文件名
     * @return 是否成功
     */
    public boolean sync(File file, String backupUrl, String originalFilename) {
        log.info("[HTTP-SYNC] ===== 开始HTTP同步 =====");
        log.info("[HTTP-SYNC] 文件: {}, 目标URL: {}, 原始文件名: {}",
                file.getAbsolutePath(), backupUrl, originalFilename);

        try {
            // 1. 构建 multipart 请求
            HttpHeaders headers = new HttpHeaders();
            headers.setContentType(MediaType.MULTIPART_FORM_DATA);

            // 2. 设置自定义请求头，标识这是同步请求
            headers.set("X-Filename", file.getName());           // UUID 文件名
            headers.set("X-Orig-Filename", originalFilename);    // 原始文件名

            // 3. 构建文件资源
            FileSystemResource fileResource = new FileSystemResource(file);
            MultiValueMap<String, Object> body = new LinkedMultiValueMap<>();
            body.add("file", fileResource);

            HttpEntity<MultiValueMap<String, Object>> requestEntity =
                    new HttpEntity<>(body, headers);

            // 4. 发送 POST 请求
            log.info("[HTTP-SYNC] 发送POST请求到备份节点...");
            String response = restTemplate.postForObject(backupUrl, requestEntity, String.class);
            log.info("[HTTP-SYNC] 同步完成, 备份节点返回: {}", response);

            return true;

        } catch (Exception e) {
            log.error("[HTTP-SYNC] 同步失败: {}", e.getMessage());
            return false;
        }
    }
}
```

**关键点：** `FileSystemResource` 是 Spring 对文件的抽象，
`RestTemplate` 会自动将其转换为 multipart/form-data 格式。
自定义请求头 `X-Filename` 和 `X-Orig-Filename` 用于传递文件名信息。

### 4.4 FileController — 修改 upload()

```java
@RestController
@Slf4j
public class FileController {

    @Autowired
    private MadfsConfigProperties config;

    @Autowired
    private MQSyncer mqSyncer;

    @Autowired
    private HttpSyncer httpSyncer;

    @PostMapping("/upload")
    public String upload(@RequestParam MultipartFile file,
                         @RequestHeader(value = "X-Filename", required = false) String xFilename,
                         @RequestHeader(value = "X-Orig-Filename", required = false) String xOrigFilename)
            throws IOException {

        log.info("[UPLOAD] ===== 收到上传请求 =====");
        log.info("[UPLOAD] 原始文件名: {}, 文件大小: {} bytes",
                file.getOriginalFilename(), file.getSize());

        // [v6.0 新增] 检测是否为同步请求
        String filename;
        String originalFilename;
        boolean needSync;

        if (xFilename != null && !xFilename.isEmpty()) {
            // 同步请求：使用请求头中的文件名
            filename = xFilename;
            originalFilename = xOrigFilename != null ? xOrigFilename : file.getOriginalFilename();
            needSync = false;  // 不再触发二次同步
            log.info("[UPLOAD] 检测到 X-Filename 头: {}", xFilename);
            log.info("[UPLOAD] 主从同步上传 => X-Orig-Filename: {}", originalFilename);
            log.info("[UPLOAD] 主从同步请求, 无需再次同步");
        } else {
            // 普通上传：生成 UUID 文件名
            filename = FileUtils.getUUIDFile(file.getOriginalFilename());
            originalFilename = file.getOriginalFilename();
            needSync = true;
            log.info("[UPLOAD] 普通上传 => 生成UUID文件名: {}, needSync=true", filename);
        }

        // [v2.0] 计算子目录、写入文件
        String subdir = FileUtils.getSubdir(filename);
        File dest = new File(config.getUploadPath() + "/" + subdir + "/" + filename);
        file.transferTo(dest);
        log.info("[UPLOAD] 文件写入完成: {}", dest.getAbsolutePath());

        // [v4.0] 构建 FileMeta、计算 MD5、写入 .meta
        FileMeta meta = new FileMeta();
        meta.setName(filename);
        meta.setOriginalFilename(originalFilename);
        meta.setSize(file.getSize());
        meta.setDownloadUrl(config.getDownloadUrl());

        if (config.isAutoMd5()) {
            String md5 = DigestUtils.md5Hex(new FileInputStream(dest));
            meta.getTags().put("md5", md5);
            log.info("[UPLOAD] 计算MD5: {}", md5);
        }

        FileUtils.writeMeta(dest, meta);

        // [v6.0 新增] 根据配置选择同步策略
        if (needSync) {
            if (config.isSyncBackup()) {
                // 尝试 HTTP 同步
                log.info("[UPLOAD] 需要同步, syncBackup=true");
                log.info("[UPLOAD] 尝试HTTP同步 => backupUrl: {}", config.getBackupUrl());

                boolean success = httpSyncer.sync(dest, config.getBackupUrl(), originalFilename);

                if (success) {
                    log.info("[UPLOAD] HTTP同步成功");
                } else {
                    // HTTP 失败，降级为 MQ
                    log.warn("[UPLOAD] HTTP同步失败, 降级为MQ异步补偿");
                    mqSyncer.sync(meta);
                }
            } else {
                // 直接走 MQ 异步同步
                log.info("[UPLOAD] 需要同步, syncBackup=false");
                log.info("[UPLOAD] 直接走MQ异步同步");
                mqSyncer.sync(meta);
            }
        }

        log.info("[UPLOAD] ===== 上传完成, 返回文件名: {} =====", filename);
        return filename;
    }

    // download() 和 meta() 方法保持不变...
}
```

## 6. 版本总结

### 学到了什么

| 知识点 | 说明 |
|--------|------|
| 同步 vs 异步 | 同步保证实时性，异步保证可用性，两者可以互补 |
| 降级策略 | 主路径失败时自动切换备用路径，提高系统鲁棒性 |
| 自定义请求头 | 用 HTTP 头传递元信息，区分不同类型的请求 |
| `RestTemplate` | Spring 的 HTTP 客户端，支持 multipart 上传 |
| 防循环设计 | 分布式系统中常见问题，需要用标识位打破循环 |

### 与 v5.0 对比

| 对比项 | v5.0 | v6.0 |
|--------|------|------|
| 同步方式 | 仅 MQ 异步 | MQ 异步 + HTTP 同步（可选） |
| 同步延迟 | 1~3 秒 | 实时（HTTP 模式） |
| 一致性 | 最终一致 | 强一致（HTTP 模式） |
| 可用性 | 高（异步不阻塞） | 中（HTTP 失败会降级） |
| 适用场景 | 通用场景 | 主从架构、实时性要求高 |

### 局限性

- 只支持一个备份节点（`backupUrl` 是单个地址），不支持多备份
- HTTP 同步是阻塞的，备份节点响应慢会拖慢上传速度
- 没有重试机制，HTTP 失败立即降级，不会重试几次
- 备份节点宕机期间，降级为 MQ，但 MQ 消息可能堆积

### 设计要点

**为什么 HTTP 失败要降级而不是直接返回错误？**

如果 HTTP 同步失败就让上传失败，用户体验很差——"我的文件明明上传成功了，为什么报错？"
降级为 MQ 异步补偿，保证了：
1. 用户上传不受备份节点影响（可用性）
2. 文件最终会同步到备份节点（最终一致性）

这是分布式系统中"可用性优先"的典型设计。

**`X-Filename` 头的命名：**

用 `X-` 前缀是 HTTP 自定义头的惯例（虽然 RFC 6648 已不推荐，但仍广泛使用）。
也可以用 `Madfs-Filename` 这样的命名，更明确表示这是 madfs 专用的头。

**为什么不用 HTTP 响应状态码区分成功/失败？**

`RestTemplate.postForObject()` 在收到非 2xx 状态码时会抛异常，
我们在 `catch` 块里统一处理为"失败"，不需要额外判断状态码。
如果需要更细粒度的错误处理（如区分 404、500、超时），
可以用 `RestTemplate.exchange()` 获取完整的 `ResponseEntity`。
