# v7.0 — 配置中心化 + 健康检查接口

## 1. 版本目标

v6.0 实现了 HTTP 同步备份，但节点状态对外不透明：运维人员无法快速知道节点是否存活、存了多少文件、当前配置是什么。

v7.0 解决两个问题：
1. **配置完整化**：`MadfsConfigProperties` 覆盖所有运行参数，启动时打印完整配置
2. **可观测性**：新增 `/health` 和 `/list` 接口，让节点状态随时可查

---

## 2. 核心功能说明

### 类比：仓库状态看板

> 想象仓库门口挂了一块电子看板，任何人扫一眼就能知道：
> - 仓库是否正常运营（status: UP）
> - 仓库地址（node）
> - 货架总数 / 已用货架数（fileCount）
> - 当前运营模式（syncBackup、autoMd5）

`/health` 就是这块看板，`/list` 是仓库的货物清单。

### 接口关系图

```
客户端
  │
  ├─ GET /health  ──► 返回节点状态 JSON（存活 + 配置摘要）
  │
  ├─ GET /list    ──► 返回文件列表（分页）
  │
  ├─ POST /upload ──► 上传文件（同 v6.0）
  └─ GET /download──► 下载文件（同 v6.0）
```

### 健康检查响应示例

```json
{
  "status": "UP",
  "node": "http://localhost:8090/download",
  "storagePath": "/Users/xxx/madfs",
  "fileCount": 42,
  "syncBackup": false,
  "autoMd5": true
}
```

## 4. 代码实现说明

### MadfsConfigProperties.java — 完整配置类

```java
@Data
@Component
@ConfigurationProperties(prefix = "madfs")
public class MadfsConfigProperties {
    // 存储根路径
    private String uploadPath = System.getProperty("user.home") + "/madfs";

    // 本节点下载地址（用于 MQ 消息去重 + FileMeta 记录）
    private String downloadUrl = "http://localhost:8080/download";

    // 备份节点上传地址（HTTP 同步用）
    private String backupUrl = "";

    // MQ 消费者组（每个节点必须唯一）
    private String group = "madfs-default";

    // 上传时是否自动计算 MD5
    private boolean autoMd5 = true;

    // 上传时是否同步 HTTP 备份
    private boolean syncBackup = false;
}
```

### FileController.java — /health 接口

```java
@GetMapping("/health")
public Map<String, Object> health() {
    Map<String, Object> result = new LinkedHashMap<>();
    result.put("status", "UP");
    result.put("node", config.getDownloadUrl());
    result.put("storagePath", config.getUploadPath());
    result.put("fileCount", FileUtils.countFiles(config.getUploadPath()));
    result.put("syncBackup", config.isSyncBackup());
    result.put("autoMd5", config.isAutoMd5());
    return result;
}
```

### FileController.java — /list 接口

```java
@GetMapping("/list")
public Map<String, Object> list(
        @RequestParam(defaultValue = "0") int page,
        @RequestParam(defaultValue = "20") int size) {

    List<String> files = FileUtils.listFiles(config.getUploadPath(), page, size);
    long total = FileUtils.countFiles(config.getUploadPath());

    Map<String, Object> result = new LinkedHashMap<>();
    result.put("total", total);
    result.put("page", page);
    result.put("size", size);
    result.put("files", files);
    return result;
}
```

### FileUtils.java — countFiles / listFiles

```java
/**
 * 统计存储目录下的数据文件数量（排除 .meta 文件）
 */
public static long countFiles(String uploadPath) {
    try {
        return Files.walk(Paths.get(uploadPath))
                .filter(Files::isRegularFile)
                .filter(p -> !p.toString().endsWith(".meta"))
                .count();
    } catch (IOException e) {
        return -1;
    }
}

/**
 * 分页列出文件名（排除 .meta 文件）
 */
public static List<String> listFiles(String uploadPath, int page, int size) {
    try {
        return Files.walk(Paths.get(uploadPath))
                .filter(Files::isRegularFile)
                .filter(p -> !p.toString().endsWith(".meta"))
                .map(p -> p.getFileName().toString())
                .sorted()
                .skip((long) page * size)
                .limit(size)
                .collect(Collectors.toList());
    } catch (IOException e) {
        return Collections.emptyList();
    }
}
```

### MadfsApplication.java — 启动日志

```java
@SpringBootApplication
public class MadfsApplication implements ApplicationRunner {

    @Autowired
    private MadfsConfigProperties config;

    public static void main(String[] args) {
        SpringApplication.run(MadfsApplication.class, args);
    }

    @Override
    public void run(ApplicationArgs args) throws Exception {
        FileUtils.init(config.getUploadPath());

        log.info("===== madfs 启动完成 =====");
        log.info("存储路径:   {}", config.getUploadPath());
        log.info("下载地址:   {}", config.getDownloadUrl());
        log.info("备份地址:   {}", config.getBackupUrl());
        log.info("消费者组:   {}", config.getGroup());
        log.info("自动MD5:    {}", config.isAutoMd5());
        log.info("HTTP备份:   {}", config.isSyncBackup());
        log.info("=========================");
    }
}
```

## 6. 版本总结

### 学到了什么

- `@ConfigurationProperties` 集中管理配置，比散落的 `@Value` 更易维护
- 健康检查接口是生产系统的标配，Kubernetes 的 liveness/readiness probe 就依赖它
- `Files.walk()` 遍历目录树简洁高效，配合 Stream API 可直接实现分页

### 与 v6.0 对比

| 维度 | v6.0 | v7.0 |
|------|------|------|
| 配置管理 | 部分字段缺失 | 完整覆盖所有参数 |
| 节点可观测性 | 无 | `/health` + `/list` |
| 启动信息 | 无 | 打印完整配置摘要 |
| 运维友好度 | 低 | 中 |

### 局限性

- `fileCount` 每次都全量遍历磁盘，文件量大时性能差（v8.0 引入块存储后可改为维护计数器）
- `/list` 没有按目录过滤，文件多时内存压力大
- 没有接入 Spring Boot Actuator，无法复用标准健康检查生态

### 设计要点

- `/health` 返回的 `status: UP` 是约定俗成的格式，与 Spring Actuator 保持一致，方便后续替换
- `fileCount` 排除 `.meta` 文件，只统计真实数据文件，避免数量虚高
- 分页参数用 `defaultValue` 提供默认值，调用方可以不传参数直接使用
