# v12.0 — 监控 + 告警

## 1. 版本目标

v11.0 实现了访问控制和配额管理，但系统运行状态仍然不透明：磁盘快满了不知道，同步失败率飙升没有告警，上传延迟变高无法感知。

v12.0 解决可观测性问题：
1. **指标采集**：通过 Micrometer 暴露业务指标（上传次数、下载次数、存储用量、同步延迟）
2. **Prometheus 集成**：标准 `/actuator/prometheus` 端点，供 Prometheus 定时抓取
3. **Grafana 看板**：可视化展示关键指标趋势
4. **告警规则**：磁盘使用率 > 80%、同步失败率 > 5% 时触发告警

---

## 2. 核心功能说明

### 类比：仓库安装传感器 + 监控大屏

> v11.0 的仓库有门禁，但没有监控摄像头和传感器。
>
> v12.0 给仓库装上全套监控系统：
> - **传感器**（Micrometer）：实时采集货物进出量、货架占用率、运输耗时
> - **数据采集器**（Prometheus）：每 15 秒来仓库抄一次传感器数据，存入时序数据库
> - **监控大屏**（Grafana）：把数据画成折线图、仪表盘，一眼看出趋势
> - **报警器**（AlertManager）：货架占用率超 80% 或运输失败率超 5% 时自动发短信

### 监控数据流

```

    │
    ├─ Micrometer 埋点
    │   ├─ Counter:   madfs_upload_total（上传次数）

    │   ├─ Timer:     madfs_upload_duration（上传耗时分布）
    │   ├─ Timer:     madfs_sync_duration（同步耗时分布）
    │   └─ Gauge:     madfs_storage_used_bytes（存储用量）
    │
    ▼
GET /actuator/prometheus
    │  （Prometheus 每 15 秒抓取）
    ▼
Prometheus（时序数据库）
    │
    ├─ 告警规则评估（每 1 分钟）
    │   ├─ disk_usage > 80% ──► AlertManager ──► 钉钉/邮件通知
    │   └─ sync_error_rate > 5% ──► AlertManager ──► 告警
    │

           ├─ 存储用量趋势图
           ├─ 同步成功率仪表盘

```

## 4. 代码实现说明

### MadfsMetrics.java — 指标注册与记录

```java
@Component
public class MadfsMetrics {

    private final Counter uploadCounter;
    private final Counter downloadCounter;
    private final Counter syncSuccessCounter;
    private final Counter syncFailedCounter;
    private final Timer uploadTimer;
    private final Timer syncTimer;

    @Autowired
    private MeterRegistry registry;

    @Autowired
    private MadfsConfigProperties config;

    public MadfsMetrics(MeterRegistry registry, MadfsConfigProperties config) {
        this.registry = registry;

        // Counter：只增不减，记录累计次数
        this.uploadCounter = Counter.builder("madfs.upload.total")
                .description("文件上传总次数")
                .tag("node", config.getDownloadUrl())
                .register(registry);

        this.downloadCounter = Counter.builder("madfs.download.total")
                .description("文件下载总次数")
                .tag("node", config.getDownloadUrl())
                .register(registry);

        this.syncSuccessCounter = Counter.builder("madfs.sync.total")
                .description("同步总次数")
                .tag("result", "success")
                .tag("node", config.getDownloadUrl())
                .register(registry);

        this.syncFailedCounter = Counter.builder("madfs.sync.total")
                .description("同步总次数")
                .tag("result", "failed")
                .tag("node", config.getDownloadUrl())
                .register(registry);

        // Timer：记录耗时分布（自动计算 P50/P95/P99）
        this.uploadTimer = Timer.builder("madfs.upload.duration")
                .description("文件上传耗时")
                .publishPercentiles(0.5, 0.95, 0.99)
                .tag("node", config.getDownloadUrl())
                .register(registry);

        this.syncTimer = Timer.builder("madfs.sync.duration")
                .description("节点同步耗时")
                .publishPercentiles(0.5, 0.95, 0.99)
                .tag("node", config.getDownloadUrl())
                .register(registry);
    }

    /**
     * 注册存储用量 Gauge（动态值，每次 Prometheus 抓取时实时计算）
     */
    @PostConstruct
    public void registerStorageGauge() {
        Gauge.builder("madfs.storage.used.bytes", config, cfg ->
                FileUtils.countUsedBytes(cfg.getUploadPath()))
                .description("存储已用字节数")
                .tag("node", config.getDownloadUrl())
                .register(registry);
    }

    public void recordUpload(long durationMs) {
        uploadCounter.increment();
        uploadTimer.record(durationMs, TimeUnit.MILLISECONDS);
    }

    public void recordDownload() {
        downloadCounter.increment();
    }

    public void recordSyncSuccess(long durationMs) {
        syncSuccessCounter.increment();
        syncTimer.record(durationMs, TimeUnit.MILLISECONDS);
    }

    public void recordSyncFailed() {
        syncFailedCounter.increment();
    }
}
```

### FileController.java — 上传/下载埋点

```java
@Autowired
private MadfsMetrics metrics;

@PostMapping("/upload")
public String upload(@RequestParam MultipartFile file, ...) throws IOException {
    long start = System.currentTimeMillis();
    try {
        // ... 原有上传逻辑 ...
        return filename;
    } finally {
        metrics.recordUpload(System.currentTimeMillis() - start);
    }
}

@GetMapping("/download")
public void download(@RequestParam String name, HttpServletResponse response) throws IOException {
    // ... 原有下载逻辑 ...
    metrics.recordDownload();
}
```

### HttpSyncer.java — 同步结果埋点

```java
@Autowired
private MadfsMetrics metrics;

public boolean sync(File file, String backupUrl, String originalFilename) {
    long start = System.currentTimeMillis();
    try {
        // ... 原有同步逻辑 ...
        metrics.recordSyncSuccess(System.currentTimeMillis() - start);
        return true;
    } catch (Exception e) {
        log.error("[HTTP-SYNC] 同步失败", e);
        metrics.recordSyncFailed();
        return false;
    }
}
```

### prometheus-rules.yaml — 告警规则

```yaml
groups:
  - name: madfs-alerts
    interval: 1m
    rules:

      # 磁盘使用率超 80%
      - alert: MadfsStorageHigh
        expr: |
          madfs_storage_used_bytes / madfs_storage_total_bytes > 0.8
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "节点 {{ $labels.node }} 存储使用率超过 80%"
          description: "当前使用率: {{ $value | humanizePercentage }}"

      # 同步失败率超 5%（最近 5 分钟）
      - alert: MadfsSyncErrorHigh
        expr: |
          rate(madfs_sync_total{result="failed"}[5m])
          /
          rate(madfs_sync_total[5m]) > 0.05
        for: 2m
        labels:
          severity: critical
        annotations:
          summary: "节点 {{ $labels.node }} 同步失败率超过 5%"
          description: "当前失败率: {{ $value | humanizePercentage }}"

      # 上传 P99 延迟超 1 秒
      - alert: MadfsUploadLatencyHigh
        expr: |
          histogram_quantile(0.99,
            rate(madfs_upload_duration_seconds_bucket[5m])
          ) > 1.0
        for: 3m
        labels:
          severity: warning
        annotations:
          summary: "节点 {{ $labels.node }} 上传 P99 延迟超过 1 秒"
          description: "当前 P99: {{ $value }}s"
```

## 6. 版本总结

### 学到了什么

- **Micrometer** 是 Java 监控指标的事实标准，与 Spring Boot Actuator 深度集成，一行代码接入 Prometheus
- **Counter vs Gauge vs Timer** 的选择：
  - Counter：只增不减的累计值（上传次数）
  - Gauge：实时快照值（存储用量、在线节点数）
  - Timer：耗时分布，自动计算分位数（P99 延迟）
- **PromQL** 的 `rate()` 函数：计算每秒增长率，是告警规则的核心，`rate(counter[5m])` 表示最近 5 分钟的平均增长率
- `publishPercentiles` 在客户端计算分位数，精度高但内存开销大；`publishPercentileHistogram` 在服务端计算，适合跨实例聚合

### 与 v11.0 对比

| 维度 | v11.0 | v12.0 |
|------|-------|-------|
| 系统可观测性 | 仅 /health 接口 | 完整指标 + 告警 |
| 性能分析 | 无 | P50/P95/P99 延迟分布 |
| 容量规划 | 手动查看 | 存储趋势图，提前预警 |
| 故障发现 | 被动（用户反馈） | 主动（告警通知） |
| 运维工具链 | 无 | Prometheus + Grafana |

### 局限性

- `Gauge` 注册的存储用量每次抓取都遍历磁盘，文件多时会影响 Prometheus 抓取性能（应改为定时更新缓存值）
- 没有实现分布式 Tracing（如 Zipkin/Jaeger），无法追踪单次请求在多节点间的完整链路
- 告警通知渠道（钉钉、邮件）需要额外配置 AlertManager，本版本只定义了规则
- Grafana 看板需要手动配置，没有提供可直接导入的 JSON 模板

### 设计要点

- 所有指标统一在 `MadfsMetrics` 类中注册，避免指标名称散落各处，便于维护和文档化
- `tag("node", downloadUrl)` 给每个指标打上节点标签，Prometheus 可以按节点聚合或对比
- 告警规则用 `for: 5m` 避免瞬间抖动触发误报，只有持续超阈值才告警
- `/actuator/prometheus` 端点应在生产环境加认证保护，避免指标数据泄露业务信息
