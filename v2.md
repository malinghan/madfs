# v2.0 — 文件上传接口

---

## 1. 版本目标

在 v1.0 骨架基础上，实现 `POST /upload` 接口，接收客户端上传的文件，生成 UUID 文件名，按哈希分桶写入本地存储，返回文件名供后续下载使用。

**本版本解决的问题：**
- 客户端如何把文件传到服务器？（multipart/form-data 上传）
- 如何给文件起一个全局唯一的名字？（UUID 命名）
- 文件存到哪个目录？（哈希分桶路由）

**本版本不涉及：** 文件下载、元数据、同步。上传后只能通过文件系统直接查看文件。

---

## 2. 核心功能说明

### UUID 文件命名：为什么不用原始文件名？

> 类比：快递公司收到包裹后，不会用寄件人写的"给妈妈的礼物"作为内部编号，而是贴上一个全球唯一的快递单号（UUID）。原始名字保留在快递单上，但内部流转全靠单号。

原始文件名存在的问题：
- 不同用户可能上传同名文件（`test.jpg`），会互相覆盖
- 文件名可能包含特殊字符、空格，导致路径解析问题
- 无法保证全局唯一性

UUID 方案：
```
原始文件名: test.jpg
生成 UUID:  3a9c1b2d-4e5f-6789-abcd-ef0123456789
保留扩展名: .jpg
最终文件名: 3a9c1b2d-4e5f-6789-abcd-ef0123456789.jpg
```

### 哈希分桶路由：文件存到哪里？

```
文件名: 3a9c1b2d-4e5f-6789-abcd-ef0123456789.jpg
         ↑↑
         前两位 = "3a"  =>  存入 ~/madfs/3a/ 目录

最终路径: ~/madfs/3a/3a9c1b2d-4e5f-6789-abcd-ef0123456789.jpg
```

### 上传完整流程

```
客户端
  │
  │  POST /upload
  │  Content-Type: multipart/form-data
  │  Body: file=<二进制数据>
  │
  ▼
FileController.upload(MultipartFile file)
  │
  ├─ 1. 获取原始文件名
  │      originalName = file.getOriginalFilename()  // "test.jpg"
  │
  ├─ 2. 生成 UUID 文件名（保留扩展名）
  │      uuid = UUID.randomUUID().toString()         // "3a9c1b2d-..."
  │      ext  = ".jpg"                               // 从原始名提取
  │      uuidName = uuid + ext                       // "3a9c1b2d-....jpg"
  │
  ├─ 3. 计算子目录
  │      subdir = uuidName.substring(0, 2)           // "3a"
  │
  ├─ 4. 构建目标路径
  │      dest = ~/madfs/3a/3a9c1b2d-....jpg
  │
  ├─ 5. 写入文件
  │      file.transferTo(dest)
  │
  └─ 6. 返回 UUID 文件名
         return "3a9c1b2d-....jpg"
  │
  ▼
客户端收到: "3a9c1b2d-4e5f-6789-abcd-ef0123456789.jpg"
```

### 扩展名提取逻辑

```
原始文件名          扩展名    UUID 文件名
test.jpg      =>   .jpg    =>  {uuid}.jpg
photo.PNG     =>   .PNG    =>  {uuid}.PNG
archive.tar.gz =>  .gz     =>  {uuid}.gz   (只取最后一个点后的部分)
noextension   =>   ""      =>  {uuid}      (无扩展名则不加)
```

## 4. 代码实现说明

### 4.1 FileUtils — 新增工具方法

```java
public class FileUtils {

    // v1.0 的 init() 方法保持不变...

    /**
     * 根据原始文件名生成 UUID 文件名，保留原始扩展名
     *
     * 示例:
     *   "test.jpg"    => "3a9c1b2d-4e5f-....jpg"
     *   "photo.PNG"   => "3a9c1b2d-4e5f-....PNG"
     *   "noext"       => "3a9c1b2d-4e5f-...."
     */
    public static String getUUIDFile(String originalFilename) {
        String uuid = UUID.randomUUID().toString();

        // 提取扩展名（最后一个点之后的部分）
        int dotIndex = originalFilename.lastIndexOf('.');
        String ext = (dotIndex >= 0) ? originalFilename.substring(dotIndex) : "";

        return uuid + ext;
    }

    /**
     * 取文件名前两位作为子目录名
     *
     * 示例:
     *   "3a9c1b2d-....jpg" => "3a"
     *   "ff001234-....txt" => "ff"
     */
    public static String getSubdir(String filename) {
        return filename.substring(0, 2);
    }
}
```

**关键点：** `lastIndexOf('.')` 而非 `indexOf('.')` — 处理 `archive.tar.gz` 这类多点文件名时，
只取最后一个点后的部分（`.gz`），符合通常的文件扩展名语义。

### 4.2 MadfsConfigProperties — 新增 uploadPath

```java
@Component
@ConfigurationProperties(prefix = "madfs")
public class MadfsConfigProperties {

    private String path;

    /**
     * 上传存储路径，默认与 path 相同
     * 单独暴露是为了后续可能区分"上传临时目录"和"最终存储目录"
     */
    public String getUploadPath() {
        return path;  // 当前版本直接返回 path
    }

    public String getPath() { return path; }
    public void setPath(String path) { this.path = path; }
}
```

### 4.3 FileController — 上传接口

```java
@RestController
@Slf4j
public class FileController {

    @Autowired
    private MadfsConfigProperties config;

    /**
     * 文件上传接口
     * POST /upload
     * Content-Type: multipart/form-data
     * 参数: file (MultipartFile)
     * 返回: UUID 文件名字符串
     */
    @PostMapping("/upload")
    public String upload(@RequestParam MultipartFile file) throws IOException {
        log.info("[UPLOAD] ===== 收到上传请求 =====");
        log.info("[UPLOAD] 原始文件名: {}, 大小: {} bytes",
                file.getOriginalFilename(), file.getSize());

        // 1. 生成 UUID 文件名
        String uuidName = FileUtils.getUUIDFile(file.getOriginalFilename());
        log.info("[UPLOAD] 生成UUID文件名: {}", uuidName);

        // 2. 计算子目录
        String subdir = FileUtils.getSubdir(uuidName);

        // 3. 构建目标文件路径
        File dest = new File(config.getUploadPath() + "/" + subdir + "/" + uuidName);
        log.info("[UPLOAD] 存储路径: subdir={}, dest={}", subdir, dest.getAbsolutePath());

        // 4. 写入文件
        file.transferTo(dest);
        log.info("[UPLOAD] 文件写入完成: {}", dest.getAbsolutePath());

        log.info("[UPLOAD] ===== 上传完成, 返回文件名: {} =====", uuidName);
        return uuidName;
    }
}
```

**关键点：** `file.transferTo(dest)` 是 Spring 提供的方法，内部处理了临时文件的移动/复制，
比手动用 `InputStream` 写入更简洁，且会自动清理临时文件。

### 5.6 完整验证脚本

```bash
#!/bin/bash
set -e

BASE_URL="http://localhost:8090"
STORAGE="$HOME/madfs"

echo "=== v2.0 上传功能验证 ==="

# 准备测试文件
echo "Hello madfs" > /tmp/v2test.txt

# 上传
echo "1. 上传文件..."
FILENAME=$(curl -s -X POST $BASE_URL/upload -F "file=@/tmp/v2test.txt")
echo "   返回文件名: $FILENAME"

# 验证文件名格式（UUID + .txt）
if [[ $FILENAME =~ ^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}\.txt$ ]]; then
  echo "   文件名格式正确 ✓"
else
  echo "   文件名格式错误 ✗: $FILENAME"
  exit 1
fi

# 验证文件存在
SUBDIR="${FILENAME:0:2}"
echo "2. 验证文件存储..."
if [ -f "$STORAGE/$SUBDIR/$FILENAME" ]; then
  echo "   文件存在于 $STORAGE/$SUBDIR/$FILENAME ✓"
else
  echo "   文件不存在 ✗"
  exit 1
fi

# 验证内容
echo "3. 验证文件内容..."
CONTENT=$(cat "$STORAGE/$SUBDIR/$FILENAME")
if [ "$CONTENT" = "Hello madfs" ]; then
  echo "   文件内容正确 ✓"
else
  echo "   文件内容错误 ✗: $CONTENT"
  exit 1
fi

echo ""
echo "=== v2.0 验证通过 ✓ ==="
```

## 6. 版本总结

### 学到了什么

| 知识点 | 说明 |
|--------|------|
| `MultipartFile` | Spring 对 multipart 上传的抽象，封装了文件内容、原始文件名、大小等信息 |
| `file.transferTo(dest)` | 将上传的临时文件移动到目标位置，比手动 IO 更简洁 |
| UUID 命名策略 | 解决文件名冲突问题，同时天然支持哈希分桶 |
| `lastIndexOf('.')` | 正确提取文件扩展名，处理多点文件名（如 `.tar.gz`） |

### 与 v1.0 对比

| 对比项 | v1.0 | v2.0 |
|--------|------|------|
| HTTP 接口 | 无 | POST /upload |
| 文件存储 | 目录已就绪，但无法写入 | 可通过 HTTP 上传文件 |
| 文件命名 | 无 | UUID + 原始扩展名 |
| 可测试性 | 只能看目录 | 可通过 curl 上传并验证 |

### 局限性

- 上传后无法通过 HTTP 下载，只能直接访问文件系统（v3.0 解决）
- 没有文件元数据记录，不知道原始文件名是什么（v4.0 解决）
- 没有文件大小限制校验（需在 `application.yml` 配置 multipart 限制）
- 没有文件类型校验，任何文件都可以上传

### 设计要点

**为什么返回 UUID 文件名而不是完整 URL？**

返回文件名（`3a9c....jpg`）而非完整 URL（`http://localhost:8090/download?name=3a9c....jpg`）
的好处是：客户端可以自由选择从哪个节点下载（在多节点场景下，任意节点都有这个文件）。
如果返回完整 URL，客户端就被绑定到特定节点了。

**`file.transferTo()` 的注意事项：**

Spring 的 `MultipartFile` 在请求处理完成后会自动清理临时文件，
所以 `transferTo()` 必须在请求处理期间调用，不能异步延迟执行。
