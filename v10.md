# v10.0 — 集群管理 + 自动节点发现

## 1. 版本目标

v9.0 的多节点同步依赖手动配置 `backupUrl`，每次新增或下线节点都要修改配置文件并重启。这在生产环境中不可接受。

v10.0 解决三个问题：
1. **节点自动注册**：启动时自动向注册中心登记，无需手动维护节点列表
2. **动态感知上下线**：节点故障自动剔除，恢复后自动重新加入
3. **智能路由**：上传时自动选择同步目标，支持轮询和一致性哈希两种策略

---

## 2. 核心功能说明

### 类比：仓库联盟自动登记系统

> 以前开新仓库要打电话通知每个老仓库（手动配置 backupUrl）。
>
> v10.0 引入"联盟总部"（注册中心 Nacos/ZooKeeper）：
> - 新仓库开业 → 自动到总部登记（节点注册）
> - 仓库关门 → 总部自动标记下线（健康检查 + 剔除）
> - 需要调货 → 问总部要最新仓库列表，按规则选一个（节点选择）
> - 任何仓库都能随时查到全网仓库状态（`GET /cluster/nodes`）

### 节点生命周期

```
节点启动
    │
    ▼
向 Nacos 注册
    │  serviceId=madfs, ip=本机IP, port=8090
    │  metadata: downloadUrl, group, version
    │
    ▼
ClusterManager 拉取节点列表
    │  订阅 Nacos 服务变更事件
    │
    ▼
节点正常运行
    │
    ├─ 上传文件 ──► NodeSelector 选择同步目标 ──► 同步
    │
    └─ 节点宕机 ──► Nacos 健康检查超时 ──► 从列表剔除
                        │
                        ▼
                   ClusterManager 收到变更通知
                        │
                        ▼
                   本地节点列表更新
```

### 一致性哈希路由

```
文件名 hash 值
    │
    ▼
哈希环（虚拟节点）
    │
    0 ──── node-A(8090) ──── node-B(8091) ──── node-C(8092) ──── 2^32
    │
    └─ hash("3a9c....jpg") = 1500
           │
           ▼
       落在 node-B 区间 ──► 同步到 node-B

优点：新增/删除节点时，只有相邻区间的文件需要迁移，其余不受影响
```

## 4. 代码实现说明

### NodeInfo.java — 节点信息模型

```java
@Data
@AllArgsConstructor
@NoArgsConstructor
public class NodeInfo {
    private String url;           // 节点下载地址，如 http://localhost:8090/download
    private String uploadUrl;     // 节点上传地址，如 http://localhost:8090/upload
    private String status;        // UP / DOWN
    private long registerTime;    // 注册时间戳
}
```

### ClusterManager.java — 节点列表管理

```java
@Component
@Slf4j
public class ClusterManager {

    @Autowired
    private DiscoveryClient discoveryClient;  // Spring Cloud 标准接口，Nacos/ZK 均实现

    @Autowired
    private MadfsConfigProperties config;

    // 本地缓存的节点列表（排除自身）
    private volatile List<NodeInfo> peerNodes = new ArrayList<>();

    /**
     * 启动后定期刷新节点列表（每 10 秒）
     * 生产环境应改为监听 Nacos 推送事件，避免轮询
     */
    @Scheduled(fixedDelay = 10_000)
    public void refreshNodes() {
        List<ServiceInstance> instances = discoveryClient.getInstances("madfs");
        String localUrl = config.getDownloadUrl();

        List<NodeInfo> nodes = instances.stream()
                .map(inst -> {
                    String url = "http://" + inst.getHost() + ":" + inst.getPort() + "/download";
                    String uploadUrl = "http://" + inst.getHost() + ":" + inst.getPort() + "/upload";
                    return new NodeInfo(url, uploadUrl, "UP", inst.getMetadata()
                            .getOrDefault("registerTime", "0") != null
                            ? Long.parseLong(inst.getMetadata().getOrDefault("registerTime", "0"))
                            : 0L);
                })
                .filter(node -> !node.getUrl().equals(localUrl))  // 排除自身
                .collect(Collectors.toList());

        this.peerNodes = nodes;
        log.info("[CLUSTER] 节点列表刷新: 共 {} 个对等节点", nodes.size());
    }

    /**
     * 获取所有对等节点（不含自身）
     */
    public List<NodeInfo> getPeerNodes() {
        return Collections.unmodifiableList(peerNodes);
    }

    /**
     * 获取所有节点（含自身），用于 /cluster/nodes 接口
     */
    public List<NodeInfo> getAllNodes() {
        List<NodeInfo> all = new ArrayList<>(peerNodes);
        all.add(0, new NodeInfo(
                config.getDownloadUrl(),
                config.getDownloadUrl().replace("/download", "/upload"),
                "UP",
                System.currentTimeMillis()
        ));
        return all;
    }
}
```

### NodeSelector.java — 选择策略接口

```java
public interface NodeSelector {
    /**
     * 从候选节点中选择同步目标
     *
     * @param nodes    可用节点列表
     * @param filename 文件名（一致性哈希策略需要用到）
     * @return 选中的节点，列表为空时返回 null
     */
    NodeInfo select(List<NodeInfo> nodes, String filename);
}
```

### RoundRobinSelector.java — 轮询策略

```java
@Component("roundrobin")
public class RoundRobinSelector implements NodeSelector {

    private final AtomicInteger counter = new AtomicInteger(0);

    @Override
    public NodeInfo select(List<NodeInfo> nodes, String filename) {
        if (nodes.isEmpty()) return null;
        int index = Math.abs(counter.getAndIncrement() % nodes.size());
        return nodes.get(index);
    }
}
```

### ConsistentHashSelector.java — 一致性哈希策略

```java
@Component("consistenthash")
public class ConsistentHashSelector implements NodeSelector {

    // 每个真实节点对应 150 个虚拟节点，减少数据倾斜
    private static final int VIRTUAL_NODES = 150;

    @Override
    public NodeInfo select(List<NodeInfo> nodes, String filename) {
        if (nodes.isEmpty()) return null;

        // 构建哈希环
        TreeMap<Integer, NodeInfo> ring = new TreeMap<>();
        for (NodeInfo node : nodes) {
            for (int i = 0; i < VIRTUAL_NODES; i++) {
                int hash = (node.getUrl() + "#" + i).hashCode();
                ring.put(hash, node);
            }
        }

        // 文件名 hash，顺时针找最近节点
        int fileHash = filename.hashCode();
        Map.Entry<Integer, NodeInfo> entry = ring.ceilingEntry(fileHash);
        if (entry == null) {
            entry = ring.firstEntry();  // 超过最大值，回到环头
        }
        return entry.getValue();
    }
}
```

### ClusterController.java — 集群状态接口

```java
@RestController
@Slf4j
public class ClusterController {

    @Autowired
    private ClusterManager clusterManager;

    @Autowired
    private MadfsConfigProperties config;

    @GetMapping("/cluster/nodes")
    public Map<String, Object> nodes() {
        List<NodeInfo> all = clusterManager.getAllNodes();
        Map<String, Object> result = new LinkedHashMap<>();
        result.put("localNode", config.getDownloadUrl());
        result.put("totalNodes", all.size());
        result.put("nodes", all);
        return result;
    }
}
```

### FileController.java — upload 改造（同步目标动态获取）

```java
// 原来：固定同步到 config.getBackupUrl()
// 现在：通过 NodeSelector 动态选择

@Autowired
private ClusterManager clusterManager;

@Autowired
@Qualifier("roundrobin")  // 或从配置读取
private NodeSelector nodeSelector;

// upload() 中同步部分替换为：
if (needSync) {
    List<NodeInfo> peers = clusterManager.getPeerNodes();
    if (peers.isEmpty()) {
        log.warn("[UPLOAD] 无可用对等节点，跳过同步");
    } else if (config.isSyncBackup()) {
        NodeInfo target = nodeSelector.select(peers, filename);
        log.info("[UPLOAD] HTTP同步目标: {}", target.getUploadUrl());
        boolean success = httpSyncer.sync(dest, target.getUploadUrl(), originalFilename);
        if (!success) {
            log.warn("[UPLOAD] HTTP同步失败，降级为MQ");
            mqSyncer.sync(meta);
        }
    } else {
        mqSyncer.sync(meta);
    }
}
```

## 6. 版本总结

### 学到了什么

- **服务注册与发现**是微服务的基础设施，Nacos/ZooKeeper 解决的核心问题是"谁在线、在哪里"
- **一致性哈希**的虚拟节点设计解决了节点数量少时的数据倾斜问题，是分布式存储的经典算法
- `DiscoveryClient` 是 Spring Cloud 的抽象接口，底层可以无缝切换 Nacos、ZooKeeper、Eureka
- `@Scheduled` 轮询刷新是简单实现，生产环境应改为事件驱动（监听 Nacos 推送），减少延迟和无效请求

### 与 v9.0 对比

| 维度 | v9.0 | v10.0 |
|------|------|-------|
| 节点配置 | 手动配置 backupUrl | 自动注册发现 |
| 扩缩容 | 需修改配置重启 | 动态感知，无需重启 |
| 同步目标选择 | 固定单节点 | 轮询 / 一致性哈希 |
| 故障处理 | 无感知 | 自动剔除下线节点 |
| 外部依赖 | 无 | Nacos 或 ZooKeeper |

### 局限性

- 引入了 Nacos 外部依赖，本地开发需要额外启动服务
- `@Scheduled` 刷新有 10 秒延迟，节点下线后短暂仍可能被选中
- 一致性哈希环在每次 `select()` 时重建，节点多时有性能开销（应缓存哈希环）
- 没有实现数据再平衡：新节点加入后，历史文件不会自动迁移到新节点

### 设计要点

- `NodeSelector` 设计为接口，通过 `@Qualifier` 注入不同实现，策略可配置切换
- 节点列表排除自身是关键：防止节点把文件同步给自己，造成无意义的本地复制
- `volatile` 修饰 `peerNodes`：定时任务线程写，请求处理线程读，保证可见性
- 集群接口 `/cluster/nodes` 是运维的重要工具，应在 `/health` 中也暴露节点数量摘要
