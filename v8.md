# v8.0 — 块存储改造（文件分块存储）

## 1. 版本目标

v7.0 的存储模型是"一个文件 = 一个磁盘文件"，存在两个问题：
1. **大文件上传不友好**：网络中断后必须从头重传
2. **存储浪费**：相同内容的文件会被完整存储多份

v8.0 将文件拆分为固定大小的块（默认 4MB）分别存储，解决：
- 支持**分块上传**，每块独立传输，失败只重传该块
- 支持 **HTTP Range 断点续传**下载
- 块级**内容寻址去重**：相同内容的块只存一份

---

## 2. 核心功能说明

### 类比：拆箱存储

> 把一个大包裹（文件）拆成多个标准尺寸的小箱子（块），每个小箱子贴上内容指纹（hash）作为编号。
>
> - 两个包裹里有相同的物品？只存一份，贴两张标签指向同一个箱子（块级去重）
> - 取件时按顺序把小箱子拼回原包裹（合并块）
> - 快递中途丢了一个箱子？只补寄那一个（断点续传）

### 存储结构变化

```
v7.0 存储结构:
~/madfs/3a/
├── 3a9c1b2d-....jpg          # 完整文件
└── 3a9c1b2d-....jpg.meta     # 元数据

v8.0 存储结构:
~/madfs/3a/
├── 3a9c1b2d-....jpg.block.0  # 第 0 块（0~4MB）
├── 3a9c1b2d-....jpg.block.1  # 第 1 块（4~8MB）
├── 3a9c1b2d-....jpg.block.2  # 第 2 块（8~10MB，最后一块可能不足 4MB）
└── 3a9c1b2d-....jpg.meta     # 元数据（含块信息）
```

### 分块上传流程

```
客户端
  │
  ├─ POST /upload/init?filename=test.jpg&size=10485760
  │       └─ 返回 uploadId（UUID）+ 块数量
  │
  ├─ POST /upload/chunk?uploadId=xxx&index=0  (body: 块0数据)
  ├─ POST /upload/chunk?uploadId=xxx&index=1  (body: 块1数据)
  └─ POST /upload/complete?uploadId=xxx
          └─ 合并块，写 .meta，返回最终文件名
```

### Range 下载流程

```
GET /download?name=3a9c....jpg
  │
  ├─ 无 Range 头 ──► 合并所有块，流式输出完整文件
  │
  └─ Range: bytes=4194304-8388607
          │
          └─ 定位到 block.1，直接输出该块数据
             响应: 206 Partial Content
                   Content-Range: bytes 4194304-8388607/10485760
```

### BlockMeta.java — 块元数据模型

```java
@Data
@AllArgsConstructor
@NoArgsConstructor
public class BlockMeta {
    private int index;       // 块序号，从 0 开始
    private String hash;     // 块内容 MD5（用于去重和校验）
    private long size;       // 块实际大小（最后一块可能小于 blockSize）
    private String path;     // 块文件相对路径
}
```

### FileMeta.java — 新增 blocks 字段

```java
@Data
public class FileMeta {
    private String name;
    private String originalFilename;
    private long size;
    private String downloadUrl;
    private Map<String, String> tags = new HashMap<>();

    // v8.0 新增：块信息（null 表示小文件，未分块）
    private List<BlockMeta> blocks;

    // 是否为分块存储
    public boolean isChunked() {
        return blocks != null && !blocks.isEmpty();

}
```

### BlockStorageService.java — 核心逻辑

```java
@Service
@Slf4j
public class BlockStorageService {

    @Autowired
    private MadfsConfigProperties config;

    // 临时存储上传中的块：uploadId -> 已上传块列表
    private final Map<String, List<BlockMeta>> uploadSessions = new ConcurrentHashMap<>();

    /**
     * 初始化分块上传，返回 uploadId 和预期块数
     */
    public Map<String, Object> initUpload(String filename, long totalSize) {
        String uploadId = UUID.randomUUID().toString();
        int blockCount = (int) Math.ceil((double) totalSize / config.getBlockSize());
        uploadSessions.put(uploadId, new ArrayList<>());

        Map<String, Object> result = new LinkedHashMap<>();
        result.put("uploadId", uploadId);
        result.put("blockCount", blockCount);
        result.put("blockSize", config.getBlockSize());
        return result;
    }

    /**
     * 上传单个块
     */
    public BlockMeta uploadChunk(String uploadId, int index, byte[] data, String uuidName) throws IOException {
        String hash = DigestUtils.md5Hex(data);

        // 内容寻址：相同 hash 的块只存一份
        String subdir = FileUtils.getSubdir(uuidName);
        String blockPath = subdir + "/" + uuidName + ".block." + index;
        File blockFile = new File(config.getUploadPath() + "/" + blockPath);

        if (!blockFile.exists()) {
            Files.write(blockFile.toPath(), data);
            log.info("[BLOCK] 写入块: index={}, hash={}, size={}", index, hash, data.length);
        } else {
            log.info("[BLOCK] 块已存在（去重）: index={}, hash={}", index, hash);
        }

        BlockMeta blockMeta = new BlockMeta(index, hash, data.length, blockPath);
        uploadSessions.get(uploadId).add(blockMeta);
        return blockMeta;
    }

    /**
     * 完成上传：写 .meta 文件，清理 session
     */
    public FileMeta completeUpload(String uploadId, String uuidName,
                                   String originalFilename, long totalSize) throws IOException {
        List<BlockMeta> blocks = uploadSessions.remove(uploadId);
        // 按 index 排序，防止乱序上传
        blocks.sort(Comparator.comparingInt(BlockMeta::getIndex));

        FileMeta meta = new FileMeta();
        meta.setName(uuidName);
        meta.setOriginalFilename(originalFilename);
        meta.setSize(totalSize);
        meta.setDownloadUrl(config.getDownloadUrl());
        meta.setBlocks(blocks);

        String subdir = FileUtils.getSubdir(uuidName);
        File dataFile = new File(config.getUploadPath() + "/" + subdir + "/" + uuidName);
        FileUtils.writeMeta(dataFile, meta);

        log.info("[BLOCK] 上传完成: name={}, blocks={}", uuidName, blocks.size());
        return meta;
    }

    /**
     * 合并所有块，输出到 OutputStream（用于完整下载）
     */
    public void mergeToStream(FileMeta meta, OutputStream out) throws IOException {
        for (BlockMeta block : meta.getBlocks()) {
            File blockFile = new File(config.getUploadPath() + "/" + block.getPath());
            Files.copy(blockFile.toPath(), out);
        }
    }

    /**
     * Range 下载：定位到对应块，输出指定字节范围
     */
    public void rangeOutput(FileMeta meta, long start, long end, OutputStream out) throws IOException {
        long blockSize = config.getBlockSize();
        int startBlock = (int) (start / blockSize);
        int endBlock = (int) (end / blockSize);

        long written = 0;
        long globalOffset = (long) startBlock * blockSize;

        for (int i = startBlock; i <= endBlock; i++) {
            BlockMeta block = meta.getBlocks().get(i);
            File blockFile = new File(config.getUploadPath() + "/" + block.getPath());
            byte[] data = Files.readAllBytes(blockFile.toPath());

            long blockStart = Math.max(0, start - globalOffset);
            long blockEnd = Math.min(data.length, end - globalOffset + 1);

            out.write(data, (int) blockStart, (int) (blockEnd - blockStart));
            written += blockEnd - blockStart;
            globalOffset += data.length;
        }
    }
}
```

### FileController.java — 分块上传接口

```java
// 1. 初始化上传
@PostMapping("/upload/init")
public Map<String, Object> initUpload(@RequestParam String filename,
                                       @RequestParam long size) {
    return blockStorageService.initUpload(filename, size);
}

// 2. 上传单块
@PostMapping("/upload/chunk")
public BlockMeta uploadChunk(@RequestParam String uploadId,
                              @RequestParam int index,
                              @RequestParam String filename,
                              @RequestBody byte[] data) throws IOException {
    return blockStorageService.uploadChunk(uploadId, index, data, filename);
}

// 3. 完成上传
@PostMapping("/upload/complete")
public FileMeta completeUpload(@RequestParam String uploadId,
                                @RequestParam String filename,
                                @RequestParam String originalFilename,
                                @RequestParam long size) throws IOException {
    return blockStorageService.completeUpload(uploadId, filename, originalFilename, size);
}
```

### DownloadController.java — Range 下载

```java
@GetMapping("/download")
public void download(@RequestParam String name,
                     @RequestHeader(value = "Range", required = false) String range,
                     HttpServletResponse response) throws IOException {

    FileMeta meta = FileUtils.readMeta(config.getUploadPath(), name);

    if (meta.isChunked()) {
        if (range != null) {
            // 解析 Range: bytes=start-end
            String[] parts = range.replace("bytes=", "").split("-");
            long start = Long.parseLong(parts[0]);
            long end = parts.length > 1 ? Long.parseLong(parts[1]) : meta.getSize() - 1;

            response.setStatus(HttpServletResponse.SC_PARTIAL_CONTENT);
            response.setHeader("Content-Range",
                    "bytes " + start + "-" + end + "/" + meta.getSize());
            response.setContentLengthLong(end - start + 1);
            blockStorageService.rangeOutput(meta, start, end, response.getOutputStream());
        } else {
            response.setContentLengthLong(meta.getSize());
            blockStorageService.mergeToStream(meta, response.getOutputStream());
        }
    } else {
        // 兼容 v7.0 及以前的非分块文件
        String subdir = FileUtils.getSubdir(name);
        File file = new File(config.getUploadPath() + "/" + subdir + "/" + name);
        response.setContentLengthLong(file.length());
        FileUtils.output(file, response.getOutputStream());
    }
}
```

## 5. 测试流程

```bash
# 准备一个大文件（10MB）
dd if=/dev/urandom of=/tmp/bigfile.bin bs=1M count=10

# 1. 初始化分块上传
INIT=$(curl -s -X POST "http://localhost:8090/upload/init?filename=bigfile.bin&size=10485760")
echo $INIT | python3 -m json.tool
# 预期:
# {
#   "uploadId": "xxxxxxxx-...",
#   "blockCount": 3,
#   "blockSize": 4194304
# }

UPLOAD_ID=$(echo $INIT | python3 -c "import sys,json; print(json.load(sys.stdin)['uploadId'])")
UUID_NAME=$(python3 -c "import uuid; print(str(uuid.uuid4()) + '.bin')")

# 2. 分块上传（3 块）
# 块 0: 0~4MB
dd if=/tmp/bigfile.bin bs=4194304 count=1 skip=0 2>/dev/null | \
  curl -s -X POST "http://localhost:8090/upload/chunk?uploadId=$UPLOAD_ID&index=0&filename=$UUID_NAME" \
  -H "Content-Type: application/octet-stream" --data-binary @-

# 块 1: 4~8MB
dd if=/tmp/bigfile.bin bs=4194304 count=1 skip=1 2>/dev/null | \
  curl -s -X POST "http://localhost:8090/upload/chunk?uploadId=$UPLOAD_ID&index=1&filename=$UUID_NAME" \
  -H "Content-Type: application/octet-stream" --data-binary @-

# 块 2: 8~10MB（最后一块）
dd if=/tmp/bigfile.bin bs=4194304 skip=2 2>/dev/null | \
  curl -s -X POST "http://localhost:8090/upload/chunk?uploadId=$UPLOAD_ID&index=2&filename=$UUID_NAME" \
  -H "Content-Type: application/octet-stream" --data-binary @-

# 3. 完成上传
curl -s -X POST "http://localhost:8090/upload/complete?uploadId=$UPLOAD_ID&filename=$UUID_NAME&originalFilename=bigfile.bin&size=10485760" \
  | python3 -m json.tool
# 预期返回 FileMeta，含 blocks 数组（3 个块）

# 4. 验证块文件存在
SUBDIR="${UUID_NAME:0:2}"
ls ~/madfs/$SUBDIR/
# 预期: xxx.bin.block.0  xxx.bin.block.1  xxx.bin.block.2  xxx.bin.meta

# 5. 完整下载并验证
curl -s "http://localhost:8090/download?name=$UUID_NAME" -o /tmp/downloaded.bin
diff /tmp/bigfile.bin /tmp/downloaded.bin && echo "完整下载一致 ✓" || echo "不一致 ✗"

# 6. Range 下载（只取第 1 块：4MB~8MB）
curl -s -H "Range: bytes=4194304-8388607" \
  "http://localhost:8090/download?name=$UUID_NAME" -o /tmp/range.bin
# 验证响应头
curl -I -H "Range: bytes=4194304-8388607" "http://localhost:8090/download?name=$UUID_NAME"
# 预期: HTTP/1.1 206 Partial Content
#       Content-Range: bytes 4194304-8388607/10485760
```

## 6. 版本总结

### 学到了什么

- **内容寻址存储**（Content-Addressable Storage）是对象存储的核心思想，块的 hash 既是校验码也是去重键
- HTTP `206 Partial Content` + `Content-Range` 是断点续传的标准协议，视频播放器、下载工具都依赖它
- `ConcurrentHashMap` 管理上传 session 是简单实现，生产环境应持久化到 Redis（防止重启丢失）
- 分块上传天然支持并发：多个块可以同时上传，互不干扰

### 与 v7.0 对比

| 维度 | v7.0 | v8.0 |
|------|------|------|
| 存储模型 | 整文件存储 | 分块存储 |
| 大文件支持 | 弱（内存压力大） | 强（流式分块） |
| 断点续传 | 不支持 | 支持（Range 请求） |
| 存储去重 | 不支持 | 块级去重 |
| 实现复杂度 | 低 | 中高 |

### 局限性

- 上传 session 存在内存中，服务重启后未完成的上传会丢失
- 块大小固定（4MB），不支持动态调整
- 没有实现块的并发上传客户端（需要客户端配合）
- Range 下载实现假设块大小均匀，最后一块特殊处理需仔细验证边界

### 设计要点

- `.meta` 文件中的 `blocks` 字段是关键：下载时完全依赖它重建文件顺序
- 兼容性：下载接口通过 `meta.isChunked()` 判断，同时支持 v7.0 的整文件和 v8.0 的分块文件
- 块文件命名 `{uuid}.block.{index}` 与原文件名绑定，避免跨文件块混淆
